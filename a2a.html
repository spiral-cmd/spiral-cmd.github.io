Known failure modes

1

WatchPaths misses atomic writes

launchd watches the inode, not the path. If Polo replaces to-cc.md atomically (write-then-rename), the inode changes and launchd never fires. No notification, no log entry, silent drop.

2

Notification is not acknowledgement

Even when the watcher fires correctly, the macOS notification can be dismissed, missed, or suppressed by Focus mode. There is no retry, no confirmation, no fallback — the message sits until Martin manually runs /csn.

3

CC is passive — no persistent session

CC only reads sermo and to-cc at session start (hook) or when Martin invokes /csn. Polo→CC messages can sit unread for hours. Observed gap today: original Codex report written ~09:45 BST, read at 17:18 UTC (~7.5h delay).

4

to-cc.md is cleared by /csn before Polo finishes writing

Race condition observed today: /csn cleared the file at 17:18:34Z, Polo wrote a second time at 17:20:00Z (38s later). That write sat unread until the next session. The watcher did fire at 17:20:55Z — but only because Polo explicitly re-sent.

Proposed fixes

Option A — Dual-channel delivery (recommended)

Every Polo→CC message goes to both to-cc.md AND sermo topic 111 with a [→CC] prefix.

  • Sermo JSONL is append-only, never cleared, always readable by /csn
  • File write becomes a convenience cache, not the channel
  • No dependency on launchd, WatchPaths, or notifications
  • Martin sees every message on Telegram as a side-effect
Eliminates failure modes 1 + 2 + 4

Option B — Poll-based launchd

Replace WatchPaths with StartInterval: 90 so launchd polls every 90s regardless of file events.

  • Atomic writes, inode replacement — irrelevant to polling
  • Notification still dismissable (failure mode 2 remains)
  • Does not fix the 7.5h passive CC gap (failure mode 3)
  • Simple change to the plist
Fixes failure mode 1 only
