Source 02 — io / General

The Baseline: What OpenClaw Actually Does

  • No native read support. GitHub issue #12756 documents it explicitly: openclaw message read returns "Action read is not supported for provider telegram."
  • Bot API ceiling. Even if a read action existed, the Bot API cannot return messages the bot itself sent — by design, to prevent loops. Gateway uptime only; no retroactive retrieval.
  • Sidecar JSONL = accepted community pattern. Middleware captures outbound messages into a local JSONL/SQLite at send time. Agents query the local DB — not the Telegram API — for context.
  • MTProto / Telethon = the advanced alternative. Agent runs as a "User" account rather than a "Bot." Full chat history, including self-sent messages. Higher complexity, risk of account flagging if too aggressive.
  • Local Bot API server doesn't help. Removes file size limits and rate constraints but does not bypass the bot self-message visibility limitation.

Source 03 — DeepSeek

A Ready-Made Skill Already Exists

  • message:sent hook is broken. Confirmed — it never fires for direct agent responses. Cannot be used for outbound logging. A message:sending pre-send hook is proposed but not yet implemented.
  • telegram-history skill by rhlsthrm (v1.1.0, Feb 2026) Key Finding — ClawHub-installable skill using Telethon/MTProto. Fetches full chat history including old messages, group topics, and forum threads invisible to bots.
  • Install: npx clawhub@latest install telegram-history. Requires api_id/api_hash from my.telegram.org. One-time 2FA login → stores local .session file. Supports: chat_id, topic_id, limit, offset-id, JSON output.
  • Session gotchas. Unclean exit can corrupt .session file. Concurrent process access fails. macOS cron has limited env — no keychain, different PATH. Always use absolute paths, activate venv explicitly.
  • tg-cli skill also exists — stores session data and history locally, less documented.
DeepSeek bottom line: The most mature, OpenClaw-integrated path is the telegram-history skill. It's purpose-built for this exact problem, avoids reinventing the wheel, and handles topic IDs, pagination, and 2FA.

Source 04 — Grok

macOS Daemon Gotchas + Cross-Framework Consensus

  • Confirms same architectural gap. Outbound CLI sends frequently miss agentId — mirror object never constructed, JSONL write skipped. Not incidental, systemic.
  • JSONL sidecar confirmed community default — manual or hook-based appends to session transcripts. Per-topic memory subdirs common in multi-topic setups.
  • telegram-history skill confirmed — openclaw skills install rhlsthrm/telegram-history is the quickest path to full topic history.
  • macOS extra gotcha: Power Nap / sleep disrupts long-polling connections. A cron job is the wrong execution model for a persistent MTProto listener. Use launchd with KeepAlive so the process restarts automatically on crash or wake.
  • Other frameworks (AutoGen, CrewAI): Same gap. CrewAI + Composio uses Bot API getUpdates polling — same visibility limits. No major framework has a native solution.
  • General pattern across all frameworks: External logger (Telethon userbot / webhook archive / bridge DB) + pre-turn context injection.
Grok summary: Ranked approaches — (1) telegram-history skill, (2) JSONL sidecar + hooks, (3) userbot channel plugin, (4) companion Telethon service. No single config flag fixes the gap; the ecosystem has mature workarounds.

Source 05 — "Beyond the Bot" Essay

Architectural Deep Dive: No Documented Examples Yet

  • No documented Telethon + OpenClaw examples exist. A comprehensive search found no end-to-end case study or open-source project showing this combination. The space is nascent — pioneering territory.
  • MTProto reader as persistent companion service (not a script) is the most powerful pattern. Maintains real-time connection; exposes a REST or gRPC API the agent queries for context.
  • Forwarding bot pattern — admin bot copies every message to a private archive channel the agent can read. Easy setup, but brittle: latency, failure risk, needs admin rights.
  • Biggest single risk: macOS daemon session + 2FA. A daemonized process that gets prompted for a 2FA code has no way to respond. Without an automated re-auth strategy, any headless solution will eventually fail.

Source 06 — Kimi

Bug Numbers, StringSession, and the SOTA Architecture

  • Bug #54186 Confirmed — openclaw message send CLI delivers to Telegram but never writes to session JSONL. Root cause: CLI omits agentId/sessionKey, so appendAssistantMessageToSessionTranscript is skipped. Affects all channels. Fix proposed 2026-03-24, status unconfirmed.
  • Feature #56593 — Proposed openclaw message inject (appends to JSONL and sends). Not yet implemented. Direct JSONL appends risk corruption while gateway is running.
  • Feature #54259 PR Ready — "Multi-agent group transcript" with working PR. Auto-logs agent responses to shared transcript; designed for exactly this Bot API limitation. Watch this.
  • Use StringSession for cron/daemon Critical — not default SQLiteSession. SQLiteSession files bind to filesystem and break across container/cron contexts. StringSession stores auth as portable base64 → store in env var or keychain.
  • Hooks remain unreliable. Issues #15389, #3945, #14210, #49765, #16596, #29203, #70928, #50126 — message_sending/message_sent hooks fragmented, partially unwired, silently fail. Do not rely on them.

Sources 15 + 16 — Notion + Email

Two Low-Effort Paths That Actually Work

  • Email channel also broken (#54186). Bug is at CLI layer, pre-channel. openclaw message send --channel email skips JSONL exactly like Telegram. No channel escape.
  • Inbound email to agent = zero-code workaround. Outbound message send uses broken CLI path. Inbound email hits the Gateway inbound pipeline — always writes to session JSONL. Route cron notifications to the agent's email address instead of using message send.
  • Agent email via OpenMail skill. clawhub install openmail. Provisions <agent>@<workspace>.openmail.sh. Caveats: SMTP latency (seconds–minutes); agent may auto-reply (token cost per notification); existing Gmail may be send-only — verify inbound is configured.
  • Notion as structured context store. ClawHub Notion skill (13.9K downloads) provides native notion-query-database, notion-create-page tools — no shelling out. Cron scripts write rows; agents query by date/topic on demand. Database beats page-append for structured filtering.
  • Notion gap: entries are not in session JSONL — agent must be explicitly asked. Not automatic context injection.
