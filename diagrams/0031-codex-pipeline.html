How Codex Works

1

Document Arrives in pdf-in/

A PDF lands in the pdf-in/ watch folder — dragged in manually, saved from a browser, or downloaded by a script.

pdf-in/ folder
2

Text Extraction

The first 2 pages are extracted using pdftotext -l 2. If fewer than 50 characters come back (scanned image PDF), the pipeline falls back to OCR via Tesseract. Only the first 2 pages are needed — title, abstract, and introduction carry the most signal for tagging.

pdftotext (primary) Tesseract OCR (fallback) first 2 pages only
3

Keyword Tagger

The extracted text is scored against codex-tags.json — a lookup table of 28 canonical tags, each with 3–15 keywords. Hit counts determine the tag. High-specificity single keywords (e.g. "Nadal" → tennis, "Kingsdale" → pratica/zee) can trigger auto-tag on one hit alone.

≥ 2 HITS Auto-tagged. Copied to codex/<tag>/ with confidence score.
< 2 HITS Low confidence. Held in codex/00-tosort/ for manual review.
4

Dry-Run Report (Google Sheet)

The pipeline never moves files automatically. Every run writes a report row to a Google Sheet: filename, proposed tag, confidence, matched keywords, proposed action. Martin reviews and approves rows. Only approved rows proceed to the apply step.

always dry-run Google Sheet review Martin approves
5

Apply — File Routed to Tag Directory

After Martin's vai, the apply script moves each file using ditto (preserves macOS Finder tags). Files route to codex/<tag>/, pratica-staging/, or 00-tosort/. A rollback manifest is written on every execute run.

ditto (xattr-safe) rollback manifest one review pass only
