OpenClaw Messaging Channels

sessions_send
Direct Synchronous
Like a phone call — sends a message directly to another agent session. Waits for response with timeout.
Best when: Target is known to be idle and you need an immediate reply.
mailbox.sqlite
Persistent Inbox
Like voicemail — survives across sessions, works when target is offline. Has dedup_key + acked_at for tracking.
Best when: Message must persist, target may be offline, or you need an audit trail.
Sermo
Telegram Broadcast / Logging
Human-visible log and broadcast via Telegram topics. Topic 111 = log entries, Topic 10 = directed exchange.
Best when: Human needs to see the message or for permanent logging.

When Each Channel Is Used

Scenario Channel Why
Polo → io (review request) sessions_send Direct, immediate, io was idle
io → Polo (approval) sessions_send Direct, immediate
CC → io (task brief) mailbox Persistence, CC may not be online when io reads
CC → agents (architectural decisions) mailbox Audit trail, survives session resets
Agent → human (log) Sermo 111 Human-visible log
Agent → agent (cross-session, target may be busy) Both sessions_send for immediate, mailbox for persistence

When Both Are Used

Use sessions_send for immediate delivery + mailbox for persistent record when:
  • Critical messages that cannot be lost
  • WP completion reports (Polo writes to mailbox + sends via sessions_send)
  • Messages to CC that need tracking via dedup_key
Note: sessions_send works reliably when target is idle. During sessions_yield, messages can get queued as "pending" — this is why critical messages go to both channels.
