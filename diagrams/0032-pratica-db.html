Pratica DB — How It Works

1

Two Scan Roots

The ingest watches two source directories. Both are scanned on every run.

pratica-mc Google Drive synced folder. Personal admin: bills, legal, banking, insurance, car, travel.
biz-cz /Volumes/4tb-a/gdrive/biz-cz. Family content: zee, kfs-cz, kanban-cz, hargreaveslansdown.
2

Daily Ingest — 11:00 London

Polo's cron walks both roots recursively via os.walk(). Each file is hashed (SHA-256). If the hash already exists in the DB the file is silently skipped — no duplicates regardless of filename or path.

daily 11:00 cron SHA-256 dedup pdf · txt · jpg · png
3

Text Extraction

PDFs are processed with pdftotext; images with Tesseract OCR. Text is chunked at 4,096 bytes for embedding. Files with no extractable text are indexed by filename and metadata only.

pdftotext Tesseract OCR 4096-byte chunks
4

Three-Layer Index

Each document is written into three parallel indexes in pratica.db.

pratica_docs Metadata: folder_path, file_name, file_type, hash, raw text. 2,458 records.
codex_fts SQLite FTS5 full-text index. Keyword search across all document text.
pratica_vec sqlite-vec vector table. Semantic search via text embeddings.
5

Search + Daily Report

Documents are searchable by keyword or semantic query via the CLI. A daily report runs at 11:15 and posts a summary to Telegram topic 2843.

pratica search <query> pratica search --folder <folder> daily report → Telegram 2843
