File-based queue at ~/.openclaw/workspace/to-cc.md. Polo and io append eval reports, WP completions, and escalations here. CC reads and clears at session start (via sermo-check.sh hook) or mid-session via /csn. Every read is archived to to-cc-log.md before clearing.

CC → CC (Session Handoff)
Persistent file at ~/.openclaw/workspace/cc-handoff.md. CC writes a session summary at the end of each session — completed work, open items, pending Martin actions, key paths. When Each Channel Is Used

Scenario Channel Why
Polo → io (review request) sessions_send Direct, immediate, io was idle
io → Polo (approval) sessions_send Direct, immediate
CC → io (task brief) mailbox Persistence, CC may not be online when io reads
CC → agents (architectural decisions) mailbox Audit trail, survives session resets
Agent → human (log) Sermo 111 Human-visible log
Polo / io → CC (eval, WP report, escalation) to-cc.md CC is passive; file queue survives until next session; archived to to-cc-log.md
CC → CC (session continuity) cc-handoff.md Persists across conversation resets; read-only at session start, never cleared
Agent → agent (cross-session, target may be busy) Both sessions_send for immediate, mailbox for persistence

When Both Are Used

Use sessions_send for immediate delivery + mailbox for persistent record when:
  • Critical messages that cannot be lost
  • WP completion reports (Polo writes to mailbox + sends via sessions_send)
  • Messages to CC that need tracking via dedup_key

CC Is Passive — File Channels Bridge the Gap

CC does not run continuously. It only reads messages at session start (via hook) or when Martin runs /csn. This makes sessions_send unreliable for Polo→CC delivery. The file-based channels solve this:
  • to-cc.md queues outbound messages from Polo/io; CC drains on next session start
  • cc-handoff.md preserves CC's own working memory across conversation resets
  • Both are read by the sermo-check.sh UserPromptSubmit hook automatically
Note: sessions_send works reliably when target is idle. During sessions_yield, messages can get queued as "pending" — this is why critical messages go to both channels. For CC specifically, always use to-cc.md rather than sessions_send, as CC has no guarantee of an active session.
